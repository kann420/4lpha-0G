# PolicyVault V4 — Redeploy + Full-Fix Runbook

> Companion to `docs/vault-v4-bug-audit.md`. This is the execution plan to fix every audited blocker + the findings that gate community use, redeploy the shared V4 infra to mainnet, verify with real (small) 0G smoke runs, and only then open V4 to the community.
> Deadline-driven. Phases A–D are local, reversible, **no funds**. Phases E–F spend **real 0G** and are the point of no return; do them only after A–D are green.

## 0. Strategy & key decisions

**0.1 Current on-chain state (verified from `.data/deployments/`):**
- `VaultRegistryV4` = `0xc41c26CE75305ED25f3371BEeb124059EE16b1E9` (mainnet, deployer `0xd7e0`).
- `ZiaLpAdapterV4` = `0x0B996674B6f937641C3a9cE993FB50677D0D81F6` (mainnet).
- Exactly **one** per-user trio exists — the deployer's own (`swap 0x64f5 / lpEntry 0xf304 / lpExit 0xA5Ce`), migrated from V3 `0xfd39`. **No community users are on V4.** → we are pre-launch; a clean redeploy strands nobody.

**0.2 Redeploy decision: FRESH registry + FRESH adapter, abandon the old trio.**
- B3 lives in `ZiaLpAdapterV4` (shared) → the adapter MUST be redeployed (new address).
- B4 lives in `PolicyVaultV4LpExit` (per-user, deployed from UI bytecode) → fixing the source + regenerating bytecode makes every *future* trio correct; the one existing trio is abandoned.
- `VaultRegistryV4` is not buggy, but it is one-shot-per-owner and the deployer's slot is already taken by the buggy trio. Deploy a **fresh registry** (~1.4KB, trivial gas) so the whole V4 namespace is clean and the deployer can re-register a fixed test trio. Old registry/adapter/trio become dead.
- **Net mainnet redeploys: 2 shared contracts (adapter + registry).** Per-user trios deploy from the UI as before (now with fixed bytecode).

**0.3 ⚠️ CRITICAL non-obvious step — regenerate the UI deploy bytecode.**
The UI deploys per-user trios from `lib/contracts/policy-vault-v4-bytecode.ts`, generated by `scripts/generate-policy-vault-v4-bytecode.mjs` from Hardhat artifacts. **After ANY `.sol` change + recompile you MUST regenerate this file**, or the UI keeps deploying the OLD buggy LpExit/Swap/LpEntry. This is Phase D and is a hard gate before redeploy.

**0.4 ⚠️ The V4 smoke script is a stub.** `scripts/lp-mainnet-v4-smoke.ts` only prints "is registered" — every `smoke:v4:*` npm script is a no-op today (audit L3). Phase F either (a) implements the smoke harness, or (b) runs the 10 paths **manually via the UI**. Given the deadline, (b) is the realistic default; (a) is SHOULD-do debt.

**0.5 Fund authority.** Owner has authorized real-0G use from the deployer wallet (`0xd7e0`) for mainnet deploy + smoke. Budget + a balance pre-check are in §Funds. Never expose the private key; deploys run through the existing hardhat `ogMainnet` network config that reads `DEPLOYER_PRIVATE_KEY` from env.

**0.6 Scope tiers** (see §1 table): **MUST** = in this redeploy (blocks basic function). **SHOULD** = before wide community rollout (robustness/security). **DEFER** = post-launch debt, recorded honestly.

## Phase A — Contract fixes (local, no funds)  ☐

**A1 — B4: pause must not block exits (`PolicyVaultV4LpExit.sol`).**
- Add modifier `onlyExecutorNotRevoked` (checks `executorRevoked` only, NOT `paused`).
- Swap the 6 exit functions (`unstakeLp:262`, `zapOut:310`, `decreaseLiquidity:365`, `collectFees:430`, `burnLp:484`, `sweepToken:534`) from `executorActive` → `onlyExecutorNotRevoked`.
- Leave `onlyOwner` recovery fns untouched (already ungated by executor state).

**A2 — B3: `liquidityOf` must survive a burned token (`ZiaLpAdapterV4.sol:378-381`).**
- Wrap `NFPM.positions(tokenId)` in try/catch; return `0` when it reverts (burned token). Apply the same guard to `positionTicks` (`:384`) if the exit post-checks read it.
- Re-verify `LpExit.zapOut:342` post-check (`liquidityOf(tokenId) != 0`) and `purgeLpNft` (`LpEntry.sol:692`) now see `0` for a fully-burned position → purge proceeds.

**A3 — M2: forbid zero-quote exits (`PolicyVaultV4LpExit.sol`).**
- In `zapOut` (`:319`) and `sweepToken` (`:551`), reject `quotedAmountOut == 0` (revert `LpInvalidMinOut`/`BadParams`) so the effective min-out floor cannot collapse to 1 wei.
- Optional stretch: thread a non-zero `amountOutMinimum` into the adapter's terminal swap.

**A4 — M1 (optional, only if low-risk): full-decrease frees full recorded native (`PolicyVaultV4LpExit.sol:408-409`).**
- When `liquidity == totalLiq`, free the entire recorded `lpNftDeployedNative[tokenId]` instead of the pro-rata floor, to avoid ghost `openLpExposure0G`. Skip if it risks destabilizing the accounting under deadline pressure — it is SHOULD, not MUST.

**A5 — Compile + size gate.**
```
npm run contracts:compile      # hardhat compile && contracts:size
```
Gate: all V4 contracts < 24576 B; LpEntry/LpExit < 23000 B. (Headroom today: LpExit 18420, LpEntry 16976 — the small additions are safe.)

## Phase B — Expand the V4 test suite (local, no funds) — THE safety net  ☐

This is what buys "works as well as V3." Do it **before** spending mainnet gas. Add to `test/policy-vault-v4.ts`:

- **B-t1 (B3 regression):** a mock NFPM that **reverts on a burned tokenId** (mirror canonical UniV3). Assert `zapOut` full-exit and `burnLp` succeed and `purgeLpNft` clears the slots. The current `MockZiaLpAdapter` returns 0 after burn and hides B3 — the new mock must revert.
- **B-t2 (B4 regression):** `setPaused(true)` then assert every exit (`unstakeLp/zapOut/decreaseLiquidity/collectFees/burnLp/sweepToken`) still succeeds; `revokeExecutor()` then assert they revert `ExecutorIsRevoked`.
- **B-t3 (M2):** `zapOut`/`sweepToken` with `quotedAmountOut==0` → revert.
- **B-t4 caps/cooldown:** re-run the fixture with **bounded** policy (perTradeCap/dailyCap/maxExposure/cooldown all finite); assert each cap + cooldown blocks the over-limit action on entries; assert exits are cap-free but `zapOut` respects min-out.
- **B-t5 reentrancy:** deploy `MaliciousZiaLpAdapterV4` in `reenter` mode; assert `nonReentrant` blocks it on an entry and an exit.
- **B-t6 malicious adapter/ERC20:** deploy `DrainingAdapter` / `MaliciousERC20`; assert the vault's balance-delta checks reject the drain.
- **B-t7 mock-rejected-on-mainnet:** construct a V4 vault with `block.chainid == 16661` + mock adapter → revert `AdapterBlocked`.
- **B-t8 executor negative:** executor cannot `withdrawNative`/`rescueToken`/`rescueNft`/select recipient (assert reverts on all three thirds).

Gate: `npx hardhat test` all green, including the new adversarial rows.

## Phase F — Mainnet smoke (REAL FUNDS, small) — the real go/no-go gate  ☐

Deploy ONE fresh trio from the deployer wallet via the UI (`createVaultV4`), then run the 10 AGENTS.md smoke paths with a **tiny** deposit (e.g. 1–2 0G). Do NOT open to community until all pass. If the smoke harness (L3) is implemented, use `smoke:v4:*`; otherwise drive each manually in the UI and record the tx hash + result:

1. ☐ Deposit into Swap vault.
2. ☐ Executor **buy** (mock adapter off / real adapter on mainnet).
3. ☐ Executor **sell** — reduces position units. (Regression for B5.)
4. ☐ LP **zapInMintLp** (entry).
5. ☐ **stakeLp**.
6. ☐ **unstakeLp** (exit → LpExit third). (Regression for B1.)
7. ☐ **zapOut** full exit — NFT burns, exposure zeros, no ghost notional. (Regression for B3.)
8. ☐ **Pause** all three thirds → confirm an executor **exit still succeeds** while paused. (Regression for B4.)
9. ☐ **revokeExecutor** → confirm entries AND exits now revert; owner recovery still works.
10. ☐ Owner **withdrawNative** from each third → funds return to owner wallet.

Bonus regressions worth running: partial `decreaseLiquidity` then re-increase (G-01 / M1 no-orphan); a partial-deploy resume in the UI (reject tx 5/7, retry → resumes, no re-deploy — H6); pause/remove a V4 agent → key disabled on all three thirds (H8).

Gate: 10/10 green on mainnet. Any red → back to the relevant phase, do NOT open to community.

## Rollback / safety

- Phases A–D are on a branch and fully reversible; nothing on-chain changes.
- If Phase F fails, the old V4 addresses still exist but are abandoned — no user funds are on them (only the deployer's own test trio). No community exposure until Phase G.
- The fresh registry/adapter means the buggy old contracts are simply never referenced again; there is no in-place upgrade to get wrong.
- Keep the audit's §6 "what is CORRECT — do not touch" in view while editing to avoid regressing the deny-by-default guarantees (recipient pinning, owner-only withdraw, struct order, V2 error ABI parity).
